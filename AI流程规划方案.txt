## Context

用户希望在现有 AI 行程规划基础上实现两个核心升级：
1. **跨模块联动**：规划路线时，联动平台的美食/非遗/数据大屏（天气+人流）模块，沿途展示天气、拥堵、美食、景点等信息
2. **地图可视化**：在行程页面嵌入交互式地图，标注路线、天气、交通、美食、非遗等 POI

同时保留原有的 P0 修复需求（TripCreate SSE 流式化、Chat 行程保存、TripDetail 新格式兼容）。

### 总体架构

```
trip_agent.generate_trip_plans_stream()
    │
    ├── 1. 开场白 (narrator LLM streaming)
    ├── 2. 对每个方案 (A/B/C 串行):
    │     ├── _generate_single_plan(LLM) → plan dict
    │     ├── IF ENABLE_ROUTE_ENRICHER:
    │     │     ├── enrich_plan(plan, interest_hints) ← asyncio.wait_for(8s)

    │     │     ├── yield trip_card (已含富化数据)
    │     │     └── yield enrichment SSE 事件 (route_weather/foods/heritages/hotels/crowd)
    │     └── yield trip_card (不含富化)
    └── 3. 收尾 → done
```

### 实施步骤

#### Step 1: 创建 Route Enricher 核心模块 (~280行)
**文件**: `backend/ai/route_enricher.py`（新建）

核心函数：
- `extract_place_names(plan)` — 从 plan 提取所有地点名
- `_geocode_location(name, db)` — 四层降级：精确匹配→子串匹配→城市回退→跳过
- `_query_nearby_*()` — 按坐标查询周边 foods/heritages/hotels/crowd（Python Haversine 过滤）
- `_query_weather_for_point()` — 查最近城市天气
- `enrich_plan(plan, interest_hints)` — 主入口，返回 enrichment + route_geo dict
- Haversine 工具函数（复用 dashboard.py 实现）

特征：
- 直接 SQLAlchemy AsyncSession 查询（不调 HTTP API）
- 500m 内地点去重 + 按 ID/key 去重
- `asyncio.gather` 并行查询 5 个数据源
- 8 秒超时保护

### 3.2 mapAdapter.js — 数据转换层 ✅ (新建, 380行)

> **这是 Phase 3 最关键的架构决策**：将所有"数据→地图 props"的转换逻辑集中到一个工具模块，MapContainer 保持纯展示。

**AI 路线数据转换**:
| 函数 | 输入 | 输出 |
|------|------|------|
| `planToMapData(plan)` | AI plan dict (含 route_geo + enrichment) | `{center, markers, routeLine, heatmapData}` |
| `waypointsToMarkers(waypoints)` | route_geo.waypoints | transport markers |
| `enrichmentToMarkers(enr)` | plan.enrichment | food/heritage/hotel/weather/crowd markers |
| `crowdToHeatmap(crowdItems)` | enrichment.crowd | heatmap data array |

**平台数据转换** (Phase 3.5 探索地图):
| 函数 | 输入 | 输出 |
|------|------|------|
| `platformDataToMapData(data)` | `{foods,heritages,events,hotels,crowd,weather}` | `{center, markers, heatmapData}` |
| `foodsToMarkers(foods)` | API 返回的美食列表 | food markers (含分类/价格/区域/热度) |
| `heritagesToMarkers(heritages)` | API 返回的非遗列表 | heritage markers (含级别/类型/传承人) |
| `eventsToMarkers(events)` | API 返回的民俗列表 | heritage markers (🎭图标, 含日期/区域) |
| `hotelsToMarkers(hotels)` | API 返回的酒店列表 | hotel markers (含星级/价格/区域) |
| `crowdLocationsToMarkers(crowd)` | API 返回的人流列表 | interest markers (含拥挤度/人数/承载量) |
| `weatherToMarkers(weather)` | API 返回的天气数据 | weather markers (含温度/天气/湿度/风力) |

**关键设计**:
- ✅ 坐标自动检测：`toLngLatArray()` 判断 `[lat,lng]` vs `[lng,lat]` 并自动翻转
- ✅ Key 兼容：`getEnrichmentArray()` 兼容 `route_` 前缀旧数据和单复数变体
- ✅ `ZOOM_THRESHOLDS` 常量导出，MapContainer 和 mapAdapter 共享
- ✅ 每个 marker 携带 `minZoom`（缩放阈值）和 `detail`（infoWindow 字段）

### 实施步骤

#### Step 1: 新建 `frontend/src/composables/useRouteDriving.js` (~300行)

核心 composable，封装所有 AMap.Driving 调用和 TMC 解析。

**模块级单例设计**:
- `_AMapInstance` + `_drivingPluginReady` — AMap 实例和 Driving 插件全局共享
- `_ensureAMapDriving()` — 异步加载 AMap + Driving 插件，幂等调用
- 每个 composable 实例独立持有 `driving` 实例和 `planRoutes` 状态

**核心方法**:
- `addPlanRoute(planId, waypoints)` — 逐方案增量添加路线
  1. 调用 `AMap.Driving.search(origin, destination, { waypoints, extensions: 'all' })`
  2. 解析 `result.routes[0].steps[]`，为每个 step 加载 TMC 数据（优先直接读取 `step.tmcs`，否则 `step.loadDetail()`）
  3. 按 `tmc.distance / step.distance * path.length` 计算分割点索引
  4. 生成分段 Polyline，每段独立 color + strokeWeight
  5. 清理旧路线（方案更新场景），添加到地图
  6. `map.setFitView()` 自适应视野
- `highlightPlan(planId)` — 选中高亮，其他半透明（Toggle: 再次点击取消）
- `resetHighlight()` — 恢复所有路线默认样式
- `clearAll()` — 移除所有路线 Polyline

**TMC 路径切割算法**:
```
step.distance (总距离) → tmc[0].distance / step.distance = ratio
→ ratio × step.path.length = 分割点索引
→ 按索引切片 path 坐标数组
→ 每段独立 AMap.Polyline，颜色由 tmc.status 决定
→ 末段剩余坐标用基线色补齐
```

**降级策略**: Driving API 失败时自动绘制虚线直线（`strokeStyle: 'dashed'`）+ 方向箭头

**响应式状态**: `planRoutes`, `highlightedPlanId`, `loading`, `hasCongestion` (computed)

## Phase 7.1：起终点标记 + 方案切换按钮 + 状态持久化 ✅ COMPLETED (2026-07-10 下午)

> **依赖**: Phase 7 (useRouteDriving.js + MapContainer 路线绘制) ✅ 全部满足
> **背景**: Phase 7 路线可以绘制但存在 3 个 UX 缺陷：无起终点标记、无可见的方案切换按钮、离开页面数据全丢
> **预计**: 3-4h / **实际**: 已完成

### 问题 1: 起终点标记

**根因**: `drawDirectLine()` 只创建 Polyline 图层，不创建起终点 Marker。且 AMap.Driving API 因 Key 权限不足 (`INVALID_USER_SCODE`) 无法使用，Phase 7 的双层实线 + TMC 分段着色方案降级为直接 Polyline 绘制。

**修复**:
- 完全移除 `AMap.Driving` 调用，改为双层实线 Polyline（outline + core）+ 方向箭头
- 新增出发城市坐标映射表 `CITY_COORDS`（深圳/广州/厦门/福州/汕头/潮州/揭阳等 20 个城市）
- `_geocodeCity()` 函数：优先查映射表 → 映射表未命中时调用 `AMap.Geocoder` 降级
- `drawDirectLine()` 新参数 `originCoord`：
  - **起点 Marker**：绿色圆点 + 城市名标签（如"深圳"），放在出发城市坐标处
  - **终点 Marker**：红色圆点 + 景点名标签，放在最后一个 waypoint 处
  - 出发城市 ≠ 第一个 waypoint 时：画虚线连接（approach line, `strokeStyle: 'dashed'`）
- Marker 通过 `_createMarkerContent()` 生成自定义 HTML 内容（圆角胶囊样式 + 左侧色条）
- Marker 生命周期：`addPlanRoute()` 创建 → `highlightPlan()` 同步透明度 → `clearAll()` 清理
- 所有 `addPlanRoute()` 调用新增第三个参数 `originCity`（从 `form.origin` 传入）

### 问题 2: 方案切换按钮

**根因**: 路线图例 `.map-route-legend` 位于 MapContainer 内 `left: 12px`，TripCreate 的悬浮侧边栏覆盖了地图左侧区域，图例被遮挡不可见。

**修复 (双方案并行)**:
- **方案 2a**: MapContainer `.map-route-legend` CSS — `left: 12px` → `right: 12px`（地图右侧不被侧边栏遮挡）
- **方案 2b**: TripCreate 底部新增 `.route-switcher` 横条（在 `.map-stats-bar` 上方）
  - 三个方案按钮：🟢 方案A · 美食 | 🟠 方案B · 文化 | 🔵 方案C · 综合
  - 当前活跃方案高亮（`rs-btn--active`）+ 阴影，非活跃方案灰色
  - 点击调用 `onLegendClick(planId)` → 同步更新 `activeResultPlanId` → 地图高亮联动

### 问题 3: 状态持久化

**根因**: TripCreate.vue 所有状态均为组件局部 `ref`/`reactive`，Vue Router 的 `page-fade` transition 使用 `mode="out-in"`，离开时组件完全销毁，返回时重新创建，状态全丢。

**修复**:
- **新建 `frontend/src/stores/trip.js`** — Pinia store，持久化到 localStorage
- **持久化内容**: `form` (出发地/天数/预算/人群/偏好/标题)、`streamPlans` (已生成的方案 A/B/C)、`streamNarration` (流式叙述)、`activeResultPlanId`
- **不持久化内容**: 地图实例、SSE 连接、DOM 引用、abort controller
- **自动保存**: 300ms 防抖 `watch([form, streamPlans, streamNarration, activeResultPlanId], ..., { deep: true })` → `localStorage.setItem()`
- **自动恢复**: `onMounted` 时从 store 读取 `form`/`streamPlans`/`streamNarration`/`activeResultPlanId` 作为初始值
- **路线重绘**: `onDiscoveryMapReady()` 中检测 `streamPlans.length > 0` → 遍历提取 waypoints → `discoveryDriving.addPlanRoute(planId, waypoints, form.origin)`
- **清理时机**: `resetForm()` / `regeneratePlan()` 调用 `tripStore.clearAll()` → 清除 localStorage + 重置所有 state
- **`hasSavedPlans`**: computed 标记，用于判断是否显示"恢复上次方案"提示
