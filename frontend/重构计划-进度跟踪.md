## 三、核心思路：Django Admin 模式

Django Admin 理念：**注册 Model + 配置 → 自动生成完整 CRUD 界面**

| Django Admin 概念 | 后端实现 | 前端实现 |
|---|---|---|
| `ModelAdmin` 类 | `AdminResource` 工厂函数 | `AdminCrud` 配置对象 |
| `list_display` | `list_display` 参数 | `columns` 数组 |
| `search_fields` | `search_fields` 生成搜索 | `searchFields` 配置 |
| `list_filter` | `filter_fields` 筛选参数 | `filters` 配置 |
| `fieldsets` | `schema_create`/`schema_update` | `formFields` 数组 |
| 自动 CRUD 路由 | 工厂生成 5 个标准端点 | 组件调用标准 API |

### 组件层级关系

```
AdminLayout（= AdminSite — 中央注册中心）
│
├── AdminSidebar（= AdminSite.app_list — 自动从注册表生成菜单）
│
└── <router-view>
    ├── AdminDashboard   — 独立页面
    ├── AdminCrud        — 通用 CRUD
    │     ├── AdminToolbar   搜索 + 筛选 + 新增
    │     ├── AdminTable     el-table + Pagination
    │     └── AdminFormDialog  创建/编辑弹窗 + ImageUploader/TagInput
    ├── AdminSettings    — 独立页面
    └── AdminDashboardData — 独立页面
```

## 四、架构决策（已确认）

| 决策项 | 结论 | 说明 |
|--------|------|------|
| AdminCrud 灵活度 | **通用组件 + slot 覆盖** | 标准场景零代码 config，特殊需求 slot 插槽覆盖 |
| 侧边栏导航 | **统一 resources 注册表** | Phase 3 一次性改好 |
| 后端工厂 hook | **预留 6 个 hook 点** | before/after_create/update/delete |
| Admin 唯一性 | **仅允许一个 admin** | 注册固定 role=user；改 admin 检查唯一性（409）；禁止删除唯一 admin |
| Hotel / FoodCategories | 本次不做 | 后续补充 |
| 图片上传 | **JSON body 传 URL 字符串** | Phase 1 新建 upload 端点 |
| 表单字段类型 | 前端手动声明 | text/select/date/textarea/image/switch/tag |
| Settings 页面 | **完全独立** | 不走 AdminResource 工厂 |
| adminApi 组织 | **手动维护 admin.js** | ~25 个方法，自动生成收益不大 |
| Select options | **声明 API 路径** | formFields 中声明 `optionsSource`，组件自行 fetch |
| NotificationBell | **暂不集成** | 无通知 API 支撑 |
| AdminResource 端点 | **支持选择性生成** | 传入 `enabled_actions` 控制哪些端点生成 |

设计要点：
- 全部 JSON body（Pydantic schema），不再用 FormData
