## 1. 技术栈（砍后版）

### 你全会、直接用

| 层级 | 技术 | 版本 | 干什么 |
|------|------|------|--------|
| 前端框架 | **Vue 3** + Vite | ≥3.4 | 所有页面和组件 |
| UI 组件库 | **Element Plus** | 最新 | 按钮/表单/表格/弹窗/布局 |
| 图表库 | **ECharts** + vue-echarts | 最新 | 数据可视化小屏（气象趋势图+人流图） |
| 状态管理 | **Pinia** | 最新 | 用户登录状态、全局数据 |
| HTTP 请求 | **Axios** | 最新 | 调后端 API |
| 后端框架 | **FastAPI** | 最新稳定版 | 全部 API 接口 |
| 数据库 | **MySQL** | 8.0+ | 所有业务数据存储 |
| ORM | **SQLAlchemy** | 2.0 | 操作数据库（async） |
| AI 框架 | **LangChain** | 最新 | RAG 问答、意图识别 |
| AI 编排 | **LangGraph** | 最新 | 行程规划多 Agent 协同（简化版） |
| LLM | **云端 API** | — | Qwen3 或 DeepSeek 的 API，不自己部署模型 |
| 定时任务 | **APScheduler** | 最新 | 每小时刷新气象/人流数据 |

### 原文档有但你不需要的（已砍 / 简化替代）

| 原文档要求 | 处理方式 | 替代方案 |
|-----------|---------|---------|
| Nuxt 3（SSR/SSG） | **砍** — 你不是内容站，不需要 SEO | **纯 Vue 3 + Vite**，全部 CSR |
| TypeScript | **砍** — 学习成本高，时间不够 | **JavaScript**，Vue `<script setup>` |
| MongoDB / Neo4j / TDengine / ClickHouse / Elasticsearch | **砍** — 5 个额外数据库，全不会 | **只用一个 MySQL**，全文搜索用 `LIKE` + 简单索引 |
| Milvus 向量数据库 | **砍** — 需要单独部署和学习 | **FAISS**（LangChain 内置，零部署） |
| Redis | **砍** — 可以先不做缓存 | 初期直接查 MySQL，性能完全够 |
| Docker / K8s / APISIX | **砍** — 运维复杂度爆炸 | 本地 `npm run dev` + `uvicorn`，部署用宝塔/Nginx |
| 阿里云全套 | **砍** — 比赛阶段不需要云原生 | 一台服务器或本地演示 |
| 推荐引擎（协同过滤/XGBoost/DeepFM/Feast） | **砍** — 你不会，且独立推荐系统是另一个领域 | **用 LangChain + LLM 做推荐**，基于规则+LLM理解用户需求 |
| vLLM 自部署推理 | **砍** — 需要 GPU 和运维能力 | **调云端 API**（Qwen3/DeepSeek 都有公开 API） |
| WebSocket 实时推送 | **砍** — 需求方明确"不用实时" | 前端每小时轮询一次气象/人流数据 |
| OAuth 2.0 微信登录 | **砍** — 需要企业资质申请 | 先用**手机号+验证码**（阿里云 SMS），微信登录后续再加 |
| 行程规划（原版 9 个 Agent 协同） | **简化** — 需求方要求保留 | 单 Agent LLM 生成方案，不做实时调整 |
| 伴手礼/盲盒/社交种草/研学（4 个独立模块） | **合并** — 需求方要求合并 | 合并为 1 个「社区推荐」页面 |
| 数据可视化大屏（全屏多图表） | **简化** — 需求方要求保留 | 精简为小屏：气象趋势 + 人流（每小时刷新） |

### 2.2 每个模块的详细功能点

#### 首页
- **Banner 轮播**：3-5 张潮汕风景/文化大图，自动轮播，点击可跳转到对应详情页
- **分类导航入口**：6 个图标卡片（美食、非遗、民俗、行程、社区、数据），点击进入对应模块
- **热门推荐**：从美食和非遗表中取 `is_recommended=TRUE` 的 Top 6，横向滚动卡片
- **搜索框**：全局搜索（输入关键词 → 搜索美食+非遗+民俗，结果聚合展示）

#### 美食推荐
- **美食列表页**：左侧分类树（粿品/海鲜/小吃/甜品/夜宵/牛肉火锅），右侧卡片网格，支持关键词搜索、价格区间筛选、排序（按热度/按时间）
- **美食/店铺详情页**：大图、名称、描述、标签、价格区间、地址（店铺含地图标注）、推荐理由
- **AI 智能推荐入口**：列表页顶部横幅按钮"不知道吃什么？让AI帮你推荐"，点击弹出对话框，输入偏好（如"想吃正宗牛肉火锅，人均100以内"），AI 返回 3-5 个推荐结果卡片

#### 非遗民俗 + 节日提醒
- **非遗列表页**：分类标签切换（国家级/省级/市级）× 类型筛选（传统戏剧/传统技艺/民俗/传统舞蹈/传统美术），卡片网格
- **非遗详情页**：富文本描述、图片画廊、视频嵌入（可选）、传承人信息、所属地区
- **民俗日历页**：月历视图，有活动的日期高亮标记，点击日期弹出当天活动列表
- **节日管理后台页**（开发者用）：表单录入节日/活动数据（名称、描述、日期、地区、类型），列表管理

#### 智能客服
- **悬浮按钮**：固定在页面右下角，点击弹出聊天窗口
- **聊天窗口**：消息列表（用户/AI 气泡）、输入框、发送按钮。AI 回复带打字机效果（逐字显示）
- **历史记录**：侧边栏显示历史会话列表（按时间排序），点击可查看历史对话
- **建议问题**：初始状态展示 3-5 个常见问题快捷按钮（"潮汕有什么好吃的？""英歌舞是什么？""工夫茶怎么喝？"）

#### 用户中心
- **注册**：手机号 + 验证码（阿里云 SMS）+ 设置密码 + 昵称
- **登录**：手机号 + 密码
- **个人中心**：头像、昵称、用户类型标签、兴趣标签编辑
- **收藏列表**：分 Tab 展示收藏的美食/非遗/活动，支持取消收藏

#### 智能行程规划
- **需求输入向导**：步骤式表单（Step1: 天数选择 | Step2: 同行人群 | Step3: 偏好权重滑块 | Step4: 确认生成）
- **行程方案详情**：按天展开的折叠面板（Day 1 / Day 2 / …），每天内容含时间线（时间 + 活动名称 + 类型标签 + 推荐理由）
- **历史行程列表**：卡片列表，显示标题、天数、创建时间
- **重新生成**：不满意可以修改参数重新生成

#### 社区推荐
- **Feed 流**：无限滚动加载动态卡片（图片+标题+正文摘要+标签+点赞/评论数+发布时间）
- **动态详情**：完整正文 + 图片画廊 + 评论区（支持一级回复）
- **发布动态**：标题、正文（富文本）、图片上传（多图，限制 9 张）、标签输入、类型选择（伴手礼/盲盒/种草/研学）
- **私信**：会话列表 + 聊天窗口（简单文本消息，不做实时推送，前端定时轮询）

#### 数据可视化小屏
- **气象趋势图**：折线图，横轴=时间（最近24小时），纵轴=温度。副轴显示湿度。地区切换下拉框（汕头/潮州/揭阳/汕尾）
- **人流图**：柱状图，横轴=地点，纵轴=人流等级（1-5），颜色深浅表示拥挤程度。每小时自动刷新数据（前端 `setInterval` 轮询 API）

### 2.3 仍然不做（比赛阶段）

- ❌ 商家端管理后台
- ❌ 政务/超级管理端
- ❌ 行程规划的实时动态调整（WebSocket/天气联动）
- ❌ 内容审核 AI（UGC 量小，人工后台看一眼即可）
- ❌ 微信小程序
- ❌ 地图路线导航（行程中不做交通规划）
- ❌ 支付/订单系统
- ❌ 推送通知（App Push / 短信营销）
- ❌ 修改密码/换绑手机号 — 注册后不可修改（比赛阶段手动改数据库即可）
- 🟡 忘记密码（后续再加）：已在 API 设计预留 `purpose: "reset_password"` 和 `POST /auth/reset-password` 接口，开发后期有余力就做

### 2.4 工作量估算

假设有 **8 周**开发时间：

| 模块 | 复杂度 | 前端 | 后端 | 预计天数 |
|------|--------|------|------|---------|
| 首页 | ⭐⭐ | 2 天 | 0.5 天 | 2-3 天 |
| 美食推荐 | ⭐⭐⭐ | 3 天 | 3 天（含 AI） | 5-6 天 |
| 非遗民俗 + 节日提醒 | ⭐⭐ | 3 天 | 3 天 | 5-6 天 |
| 智能客服 | ⭐⭐⭐ | 3 天 | 4 天（含 RAG+SSE） | 6-7 天 |
| 用户中心 | ⭐⭐ | 2 天 | 2 天 | 3-4 天 |
| 智能行程规划 | ⭐⭐⭐ | 2 天 | 4 天（含 LangGraph） | 5-6 天 |
| 社区推荐 | ⭐⭐⭐ | 4 天 | 4 天 | 7-8 天 |
| 数据可视化小屏 | ⭐⭐ | 2 天 | 1 天 | 2-3 天 |

**总计：约 6-7 周**核心开发 + 1 周联调/Bug修复 = **8 周**

> 比 v1.0 多了 2 周，因为需求方要求保留行程规划、社区推荐和大屏。但这些模块全部做了减法，复杂度可控。

## 4. 数据库设计

### 4.0 数据库设计原则

- 所有表使用 **InnoDB** 引擎，字符集 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`
- 主键均为 `INT AUTO_INCREMENT`，业务唯一键单独建唯一索引
- 外键在应用层保证完整性（不使用数据库级 FOREIGN KEY，避免级联问题）
- JSON 字段用于灵活标签/偏好存储（MySQL 8.0 原生支持 JSON 查询）
- 所有查询频繁的筛选字段建普通索引，时间字段建索引用于排序

### 5.3 认证流程详解

#### 验证码发送流程（含防刷 + 人机验证）
```
用户输入手机号 + 完成图形验证码 → POST /api/v1/auth/send-code {phone, captcha_token, purpose}
→ 后端校验图形验证码 → 校验频率限制（见下方）→ 生成6位随机验证码
→ 调阿里云SMS发送 → 存入 MySQL sms_codes 表（phone + code + expires_at + used）
→ 返回 {message: "验证码已发送", expire_seconds: 300}
```

**验证码存储方案**：存入 MySQL `sms_codes` 表（而非内存缓存），避免多进程（workers>1）时缓存不共享导致验证失败。该表定期清理过期记录。

**验证码频率限制**（防恶意刷 SMS 费用）：
| 限制维度 | 限制值 | 实现方式 |
|---------|--------|---------|
| 同一手机号 | 60 秒内仅 1 次 | `sms_codes` 表查询最近记录 |
| 同一 IP | 每小时最多 10 次 | 内存计数器（或 Redis） |
| 同一手机号 | 每天最多 5 次 | `sms_codes` 表按 phone + date 计数 |

**图形验证码**：`send-code` 接口需传入 `captcha_token`（前端滑块/点选验证码的校验 token）。后端向验证码服务（如阿里云验证码）校验通过后才发 SMS。比赛阶段可先用简单的后端生成图片验证码方案。

**purpose 参数**：区分场景 — `"register"`（校验手机号未注册）、`"reset_password"`（校验手机号已注册）。防止对已注册用户发注册验证码（浪费短信费 + 骚扰用户）。

#### 注册流程
```
用户输入手机号+图形验证码 → POST /api/v1/auth/send-code {phone, captcha_token, purpose:"register"}
→ 后端校验手机号未注册 → 通过防刷检查 → 发SMS
→ 用户输入验证码+密码+昵称 → POST /api/v1/auth/register {phone, code, password, nickname}
→ 后端校验验证码（从MySQL查，检查未使用且未过期，校验后标记used=1）
→ 校验密码强度（至少6位，建议数字+字母）→ bcrypt哈希密码 → 写入users表 → 返回JWT Token
```

#### 登录流程（含失败锁定）
```
用户输入手机号+密码 → POST /api/v1/auth/login {phone, password}
→ 后端检查登录失败次数（同一手机号，最近15分钟内 ≥5次 → 拒绝，返回E1009）
→ 查users表 → bcrypt校验密码
→ 失败：记录失败日志（phone + ip + time），返回E1003
→ 成功：清除失败记录 → 生成JWT(含user_id, role, 7天过期)
→ 返回 {token, user: {id, phone, nickname, avatar_url}}
```

**登录失败锁定**：同一手机号 15 分钟内连续失败 5 次 → 锁定 15 分钟。加错误码 `E1009`："账号已临时锁定，请15分钟后再试"。失败计数器可存 `sms_codes` 表扩展或单独 `login_attempts` 表。

#### 登出说明（JWT 固有风险）
```
POST /api/v1/auth/logout → 客户端删除 token 和 userStore 状态即可
```

> ⚠️ **风险标注**：JWT 是无状态的，客户端删除 token 后，**服务端仍认为该 token 在过期前有效**。如果 token 在登出前已被窃取，攻击者可在过期前继续使用。比赛阶段可接受（7天过期 + 无敏感交易），生产环境需加 token 黑名单（Redis/MySQL 存储已登出 token）。

#### JWT 校验中间件
```python
# 所有需要登录的接口，通过 Depends(get_current_user) 注入
from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer

security = HTTPBearer()

async def get_current_user(
    credentials = Depends(security),
    db = Depends(get_db)
):
    payload = decode_jwt(credentials.credentials)  # 解析 JWT
    user = await db.get(User, payload["user_id"])
    if not user:
        raise HTTPException(status_code=401, detail="用户不存在")
    if user.is_disabled:
        raise HTTPException(status_code=403, detail="账号已被禁用")
    return user

async def require_admin(current_user = Depends(get_current_user)):
    """管理员权限校验"""
    if current_user.role != "admin":
        raise HTTPException(status_code=403, detail="需要管理员权限")
    return current_user
```

```python
# backend/ai/llm.py
from langchain_openai import ChatOpenAI
import os

def get_llm(temperature=0.7, streaming=False):
    """
    统一的 LLM 工厂函数。
    通过环境变量切换 Qwen3 / DeepSeek。
    """
    provider = os.getenv("LLM_PROVIDER", "qwen")  # qwen | deepseek

    if provider == "qwen":
        # 阿里云百炼 API（OpenAI 兼容格式）
        return ChatOpenAI(
            model=os.getenv("QWEN_MODEL", "qwen-plus"),  # qwen-turbo / qwen-plus / qwen-max
            api_key=os.getenv("DASHSCOPE_API_KEY"),
            base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
            temperature=temperature,
            streaming=streaming,
            max_tokens=4096,
        )
    elif provider == "deepseek":
        # DeepSeek API（OpenAI 兼容格式）
        return ChatOpenAI(
            model=os.getenv("DEEPSEEK_MODEL", "deepseek-chat"),
            api_key=os.getenv("DEEPSEEK_API_KEY"),
            base_url="https://api.deepseek.com/v1",
            temperature=temperature,
            streaming=streaming,
            max_tokens=4096,
        )
    else:
        raise ValueError(f"Unknown LLM provider: {provider}")
```

**模型选择建议**：

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 智能客服（日常问答） | `qwen-turbo` / `deepseek-chat` | 速度快、便宜、够用 |
| 美食推荐（语义理解） | `qwen-plus` | 语义理解能力好 |
| 行程规划（复杂生成） | `qwen-max` / `deepseek-chat` | 需要较强推理+结构化输出 |

> **成本估算**：以 Qwen3 定价，智能客服每条问答约消耗 500-1000 token（输入+输出），按 `qwen-turbo` 定价（输入 0.0008元/千token，输出 0.002元/千token），**一条对话约 0.002-0.005 元**。比赛期间按 1000 次对话算，约 2-5 元。阿里云百炼新用户有 100 万 token 免费额度，比赛阶段完全够用。

# backend/ai/vector_store.py
from langchain_community.vectorstores import FAISS
from langchain_core.documents import Document
import json

def build_knowledge_base(embedding_model):
    """从 JSON 知识库构建 FAISS 向量索引"""
    documents = []

    # 加载美食知识
    with open("backend/ai/knowledge/food_knowledge.json", "r", encoding="utf-8") as f:
        food_data = json.load(f)
    for item in food_data:
        doc = Document(
            page_content=f"{item['name']}: {item['description']}",
            metadata={"type": "food", "name": item["name"], "category": item.get("category", "")}
        )
        documents.append(doc)

    # 加载文化知识
    with open("backend/ai/knowledge/culture_knowledge.json", "r", encoding="utf-8") as f:
        culture_data = json.load(f)
    for item in culture_data:
        doc = Document(
            page_content=f"{item['name']}: {item['description']}",
            metadata={"type": "culture", "name": item["name"], "category": item.get("category", "")}
        )
        documents.append(doc)

    # 构建 FAISS 索引
    vector_store = FAISS.from_documents(documents, embedding_model)
    # 持久化到磁盘，避免每次启动重新构建
    vector_store.save_local("backend/ai/faiss_index")
    return vector_store

def load_knowledge_base(embedding_model):
    """加载已有的 FAISS 索引（启动时调用）"""
    try:
        return FAISS.load_local(
            "backend/ai/faiss_index",
            embedding_model,
            allow_dangerous_deserialization=True
        )
    except:
        # 首次启动，构建索引
        return build_knowledge_base(embedding_model)
```

> **重要**：FAISS 索引在 FastAPI 启动时加载到内存（`@app.on_event("startup")`），作为全局单例。不用每次请求都重新加载。

### 6.4 智能客服（LangChain RAG + SSE 流式）

**原理**：用户问问题 → 从本地知识库检索相关内容 → 拼接 Prompt → LLM 生成回答 → 流式返回前端（打字机效果）

```python
# backend/ai/cs_agent.py 核心逻辑
from langchain.chains import create_retrieval_chain
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
import json, asyncio
from fastapi.responses import StreamingResponse

# 1. 从持久化目录加载 FAISS 索引（启动时加载一次）
# 2. FAISS.as_retriever(search_kwargs={"k": 3}) → 每次检索返回 Top-3 相关文档
# 3. 拼接 System Prompt + 检索到的知识 + 对话历史 + 用户问题
# 4. LLM 生成 → 流式返回（SSE: Server-Sent Events）

async def chat_stream(session_id: str, user_message: str, user_id: int | None, db, vector_store, llm):
    """
    SSE 流式返回 AI 客服回复。
    
    流程：
    1. 检索相关知识（Top-3）
    2. 拼接 Prompt
    3. llm.stream() 逐 token 输出 → yield SSE 事件
    4. 保存完整对话到 chat_messages 表
    """
    # Step 1: 检索相关知识
    retriever = vector_store.as_retriever(search_kwargs={"k": 3})
    docs = retriever.invoke(user_message)
    knowledge_context = "\n".join([d.page_content for d in docs])

    # Step 2: 拼接 Prompt
    prompt = ChatPromptTemplate.from_messages([
        ("system", """你是一个潮汕文化宣传助手，名叫"潮小文"。请根据以下知识库回答用户问题。
        
        知识库内容：
        {context}
        
        要求：
        - 如果知识库有相关信息，基于知识库回答，不要编造
        - 如果知识库没有，诚实告知并在你的常识范围内友好回复
        - 回答简洁，每次不超过 300 字
        - 语气亲切、有潮汕地方特色（偶尔用"呐""哩"等语气词）
        - 推荐美食时，给出具体店名和人均价格
        """),
        ("human", "{input}"),
    ])

    # Step 3: 流式生成
    chain = prompt | llm
    full_response = ""

    async def event_stream():
        nonlocal full_response
        # 先发送检索到的参考来源
        sources = [{"name": d.metadata["name"], "type": d.metadata["type"]} for d in docs]
        yield f"data: {json.dumps({'type': 'sources', 'sources': sources}, ensure_ascii=False)}\n\n"

        # 逐 token 流式输出
        async for chunk in chain.astream({"context": knowledge_context, "input": user_message}):
            token = chunk.content if hasattr(chunk, 'content') else str(chunk)
            if token:
                full_response += token
                yield f"data: {json.dumps({'type': 'token', 'content': token}, ensure_ascii=False)}\n\n"

        # 发送完成信号
        yield f"data: {json.dumps({'type': 'done', 'session_id': session_id}, ensure_ascii=False)}\n\n"

        # Step 4: 保存到数据库（非阻塞）
        # ... 写入 chat_messages 表 ...

    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**SSE 输出示例**（前端接收到的原始数据流）：
```
data: {"type": "sources", "sources": [{"name": "牛肉火锅", "type": "food"}, {"name": "粿品", "type": "food"}]}

data: {"type": "token", "content": "呐"}

data: {"type": "token", "content": "，"}

data: {"type": "token", "content": "潮"}

data: {"type": "token", "content": "汕"}

data: {"type": "token", "content": "最"}

data: {"type": "token", "content": "..."}

data: {"type": "done", "session_id": "abc-123-def"}
```

**前端接收 SSE 代码要点**：
```javascript

### 6.5 美食推荐（LangChain Chain）

**原理**：用户说"我想吃正宗的牛肉火锅，不要太贵的" → LLM 理解意图 + 匹配数据库中的美食 → 返回推荐结果

```python
# backend/ai/food_recommend.py 核心逻辑
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel

class FoodRecommendation(BaseModel):
    food_id: int
    name: str
    reason: str
    score: float

class RecommendResult(BaseModel):
    recommendations: list[FoodRecommendation]
    summary: str

async def recommend_foods(user_preference: str, foods_from_db: list[dict], llm):
    """
    用户输入偏好 + 数据库候选美食 → LLM 排序推荐 → 返回 Top-5

    Args:
        user_preference: 用户自然语言偏好描述
        foods_from_db: 从 MySQL 查出的候选美食列表（取前 50 条热门）
        llm: ChatOpenAI 实例
    """
    parser = JsonOutputParser(pydantic_object=RecommendResult)

    prompt = ChatPromptTemplate.from_messages([
        ("system", """你是一个潮汕美食推荐专家。根据用户的偏好描述，从候选美食列表中推荐最合适的。

        候选美食列表：
        {candidates}

        要求：
        - 从候选列表中挑选 3-5 个最匹配的美食/店铺
        - 每个推荐给出具体的推荐理由（结合用户偏好）
        - 打分 score：0.0-1.0（越高越推荐）
        - 生成一段 50 字以内的推荐总结
        - 严格输出 JSON 格式

        {format_instructions}
        """),
        ("human", "{preference}"),
    ])

    # 候选列表转为 JSON 字符串供 LLM 参考
    candidates_str = json.dumps(foods_from_db, ensure_ascii=False, indent=2)

    chain = prompt | llm | parser
    result = await chain.ainvoke({
        "candidates": candidates_str,
        "preference": user_preference,
        "format_instructions": parser.get_format_instructions(),
    })
    return result
```

> **性能优化**：先从 MySQL 查 Top-50 热门美食作为候选池（走 `is_recommended` + `view_count` 排序），再送给 LLM 筛选。不要把整个 foods 表塞给 LLM（太慢太贵）。

### 6.6 🆕 行程规划 Agent（LangGraph 简化版）

**原理**：用户填表单（天数/人群/偏好）→ LangGraph 定义的简单 StateGraph → TripAgent 节点调用 LLM → 生成 JSON 格式行程方案

```python
# backend/ai/trip_agent.py 核心逻辑
from langgraph.graph import StateGraph, END
from typing import TypedDict, Optional
from langchain_core.prompts import ChatPromptTemplate
import json

class TripState(TypedDict):
    days: int                              # 行程天数（1-7）
    crowd_type: str                        # solo / couple / family / friends
    preferences: dict                      # {"food": 0.8, "heritage": 0.5, "nature": 0.3}
    title: str                             # 行程标题
    plan: Optional[dict]                   # AI 生成的行程方案（初始为 None）
    error: Optional[str]                   # 错误信息（初始为 None）

def make_trip_agent_node(llm):
    """创建 TripAgent 节点函数"""

    async def trip_agent_node(state: TripState) -> TripState:
        """调用 LLM 生成行程方案"""

        # 构建具体的 Prompt
        prompt_text = f"""你是一个潮汕旅游规划专家。请为以下用户规划 {state['days']} 天潮汕旅游行程。

        用户信息：
        - 行程标题：{state['title']}
        - 同行人群：{state['crowd_type']}
        - 偏好权重（0-1，越高越重视）：{json.dumps(state['preferences'], ensure_ascii=False)}

        行程要求：
        - 每天安排 3-5 个活动
        - 合理分配时间（早中晚）
        - 根据偏好权重调整活动类型比例
        - 考虑不同人群需求（如带老人减少步行、情侣增加浪漫景点、家庭增加互动体验）
        - 每个活动给出推荐理由
        - 活动类型包括：food（美食）、heritage（非遗文化）、scenery（自然风光）、shopping（购物/伴手礼）、rest（休息）

        严格输出以下 JSON 格式（不要输出任何其他内容）：
        {{{{
          "title": "行程标题",
          "summary": "50字以内的行程特色总结",
          "tips": ["出行建议1", "出行建议2"],
          "days": [
            {{{{
              "day": 1,
              "title": "Day 1 主题",
              "activities": [
                {{{{
                  "time": "09:00",
                  "name": "活动名称",
                  "type": "food",
                  "description": "50字描述",
                  "reason": "推荐理由"
                }}}}
              ]
            }}}}
          ]
        }}}}
        """

        try:
            # 调用 LLM
            response = await llm.ainvoke(prompt_text)
            # 解析 JSON（LLM 可能输出带 markdown 格式的 JSON）
            content = response.content
            # 去掉可能的 markdown 代码块标记
            if "```json" in content:
                content = content.split("```json")[1].split("```")[0]
            elif "```" in content:
                content = content.split("```")[1].split("```")[0]

            plan = json.loads(content.strip())
            state["plan"] = plan
            state["error"] = None
        except json.JSONDecodeError as e:
            state["plan"] = None
            state["error"] = f"AI 生成的行程格式有误，请重试: {str(e)}"
        except Exception as e:
            state["plan"] = None
            state["error"] = f"AI 服务调用失败: {str(e)}"

        return state

    return trip_agent_node

# 定义最简单的 StateGraph（未来可扩展为多 Agent 协同）
def create_trip_graph(llm):
    """创建行程规划 StateGraph"""
    graph = StateGraph(TripState)

    # 目前只有一个 trip_agent 节点
    trip_node = make_trip_agent_node(llm)
    graph.add_node("trip_agent", trip_node)

    # 起点 → trip_agent → 结束
    graph.set_entry_point("trip_agent")
    graph.add_edge("trip_agent", END)

    return graph.compile()

# 在 FastAPI 启动时创建全局实例
# trip_app = create_trip_graph(llm)
```

router.beforeEach((to, from, next) => {
  const userStore = useUserStore()
  // 浏览类页面公开（不包创建/编辑/私信等操作页）
  const publicPages = [
    '/', '/foods', '/heritages', '/festival', '/dashboard',
    '/login', '/register', '/chat', '/community'  // community Feed流+帖子详情公开
  ]
  const isPublicPage = publicPages.some(p => to.path.startsWith(p))
  // 需要认证：不在公开列表 且 未标记 requiresAuth: false
  const needsAuth = to.meta.requiresAuth !== false && !isPublicPage

  if (needsAuth && !userStore.isLoggedIn) {
    next('/login?redirect=' + to.fullPath)  // 记住来源，登录后跳回
  } else if (to.meta.requiresAdmin && userStore.user?.role !== 'admin') {
    next('/')  // 非管理员访问后台 → 回首页
  } else {

  }

### 7.3 页面状态规范（每个页面统一三态）

每个数据页面**必须**处理三种状态：

| 状态 | UI 表现 | 触发条件 |
|------|---------|---------|
| **加载中** | 骨架屏或 `v-loading` 遮罩 | 数据请求进行中 |
| **空数据** | `<EmptyState>` 组件 + 引导文案 | 请求成功但数据为空 |
| **错误** | Element Plus `<el-alert>` 错误提示 + 重试按钮 | 请求失败（网络/服务端错误） |

> **原则**：用户看到的每个页面，不允许出现"白屏"或"一直转圈没反馈"。

| 写 Prompt 模板 | "帮我写一个潮汕美食推荐的 System Prompt，要求推荐有理有据" | AI 生成 Prompt，你微调 |
| 写知识库 JSON | "帮我写 20 条潮汕非遗知识条目，每条 150 字，包含名称、类别、描述、观赏建议" | AI 生成初稿，你校对 |

| 写 ECharts 配置 | "帮我写一个 ECharts 折线图配置，横轴是时间，纵轴是温度，带数据缩放" | 直接可用 |

### 9.2 哪些活必须你自己做

| 任务 | 为什么 AI 帮不了 |
|------|-----------------|
| 调试报错 | AI 只能给建议，不能看你的浏览器控制台/终端 |
| 设计数据库关系 | 需要你理解业务，AI 可能设计出冗余结构 |
| 决定功能取舍 | AI 会建议"全都要"，你来砍 |
| API 联调 | 前端调后端时的细节问题 AI 排查不了 |
| 最终验收 | 体验好不好只有人知道 |
| 安全配置 | JWT 密钥生成、SQL 注入防护、CORS 白名单 —— AI 可能给不安全配置 |
| 知识库内容准确度 | AI 生成的知识可能有事实错误（"潮汕"写错成"潮州"等），你本地人才能纠错 |

### 9.3 用 AI 的正确姿势

```
❌ 错误：帮我写一个潮汕文化宣传平台的前端
   → AI 给你一个巨大的不完整项目，你接不住

✅ 正确：帮我写一个 Vue 3 美食列表页面，用 Element Plus 的 Table，
   支持按分类筛选和关键词搜索，数据从 FastAPI 的 /api/v1/foods 获取
   → AI 给你一个可直接用的文件，你放进项目就能跑
```

**原则**：一次只让 AI 做一件事，你要能看懂 AI 生成的代码。

### 9.4 AI 开发工作流（每做一个功能的标准流程）

```
第1步：你写清楚需求（5 分钟）
  → 在对话里描述：要什么页面、什么组件、什么数据、什么交互

第2步：AI 生成代码（1 分钟）
  → AI 输出一个或多个文件的完整代码

第3步：你复制粘贴进项目（2 分钟）
  → 放对目录位置，检查 import 路径是否正确

第4步：跑起来看效果（5 分钟）
  → npm run dev / uvicorn，刷新浏览器看有没有报错

第5步：微调修正（10 分钟）
  → 样式不对 → 让 AI 调 CSS
  → 逻辑不对 → 告诉 AI 哪里不对，让它改
  → 报错了 → 把错误信息贴给 AI

一个功能 ≈ 20-30 分钟搞定
```

## 10. 部署方案（最简版）

### 10.1 比赛演示环境：本地或一台云服务器

```
┌──────────────────────────────────────────┐
│  一台 Linux 服务器（2核4G 够用）           │
│  OS: Ubuntu 22.04 / CentOS 7             │
│                                          │
│  Nginx (端口 80 / 443)                    │
│  ├── /            → Vue 静态文件 (dist/)   │
│  └── /api/v1/*   → FastAPI (端口 8000)    │
│  ├── /uploads/*   → 用户上传图片目录       │
│                                          │
│  FastAPI (uvicorn, 端口 8000)             │
│  ├── workers=2 (2核够用)                  │
│  └── 由 systemd 管理（自动重启）           │
│                                          │
│  MySQL 8.0 (端口 3306)                    │
│  └── 数据 + 索引 约 200MB                 │
└──────────────────────────────────────────┘
```

### 10.2 环境安装（AI 可以帮你写每一步的命令）

```bash
# ========== 服务器基础环境 ==========

# 1. 更新系统
sudo apt update && sudo apt upgrade -y    # Ubuntu
# sudo yum update -y                       # CentOS

# 2. 安装 Python 3.12
sudo add-apt-repository ppa:deadsnakes/ppa -y
sudo apt install python3.12 python3.12-venv python3.12-dev -y

# 3. 安装 MySQL 8.0
sudo apt install mysql-server-8.0 -y
sudo systemctl enable mysql
sudo systemctl start mysql
# 安全初始化：sudo mysql_secure_installation

# 4. 安装 Nginx
sudo apt install nginx -y

# 5. 安装 Node.js 22 (用于前端构建)
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install nodejs -y

# ========== 项目部署 ==========

# 6. 创建项目目录
sudo mkdir -p /var/www/chaoshan-platform
sudo chown $USER:$USER /var/www/chaoshan-platform

# 7. 克隆/拷贝项目（或直接 scp 上传）
# git clone ... /var/www/chaoshan-platform
# 或 scp -r ./chaoshan-platform user@server:/var/www/

# 8. 后端虚拟环境
cd /var/www/chaoshan-platform/backend
python3.12 -m venv venv
source venv/bin/activate

# 9. 配置环境变量
cp .env.example .env

### 10.4 Nginx 配置（`/etc/nginx/sites-available/chaoshan`）

```nginx
server {
    listen 80;
    server_name 你的域名或IP;

    # 前端静态文件
    location / {
        root /var/www/chaoshan-platform/frontend/dist;
        index index.html;
        try_files $uri $uri/ /index.html;  # Vue Router History 模式
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE 流式需要关闭缓冲
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 120s;  # AI 回复可能较慢
    }

    # 用户上传图片
    location /uploads/ {
        alias /var/www/chaoshan-platform/uploads/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
        root /var/www/chaoshan-platform/frontend/dist;
        expires 7d;
        add_header Cache-Control "public, immutable";
    }
}
```

启用配置：
```bash
sudo ln -s /etc/nginx/sites-available/chaoshan /etc/nginx/sites-enabled/
sudo nginx -t           # 测试配置
sudo systemctl reload nginx
```

### 10.5 systemd 服务（`/etc/systemd/system/chaoshan-backend.service`）

```ini
[Unit]
Description=Chaoshan Platform Backend
After=network.target mysql.service

[Service]
Type=simple
User=www-data
WorkingDirectory=/var/www/chaoshan-platform/backend
EnvironmentFile=/var/www/chaoshan-platform/backend/.env
ExecStart=/var/www/chaoshan-platform/backend/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8000 --workers 2
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启动服务：
```bash
sudo systemctl daemon-reload
sudo systemctl enable chaoshan-backend
sudo systemctl start chaoshan-backend
sudo systemctl status chaoshan-backend  # 检查状态
```

## 11. 安全与测试策略

### 11.1 安全清单（比赛阶段够用）

| 检查项 | 措施 | 优先级 |
|--------|------|--------|
| SQL 注入 | 使用 SQLAlchemy ORM 参数化查询，不用原生 SQL 拼接 | 🔴 必须 |
| XSS 攻击 | 前端输出用户内容时用 `v-text` / `{{ }}` 自动转义 | 🔴 必须 |
| 密码存储 | bcrypt 哈希，不存明文 | 🔴 必须 |
| 密码强度 | 注册时校验最少 6 位，建议数字+字母组合 | 🔴 必须 |
| JWT 安全 | 密钥从环境变量读取，设过期时间（7天），HTTPS 传输 | 🔴 必须 |
| 登录防暴力破解 | 同一手机号 15 分钟内失败 5 次锁定 15 分钟 | 🔴 必须 |
| 短信防刷 | 同一手机号 60s/次 + IP 10次/小时 + 单号 5次/天 | 🔴 必须 |
| 收藏/私信/行程权限 | 所有"仅本人"操作校验 `user_id == current_user.id` | 🔴 必须 |
| CORS 配置 | 只允许前端域名跨域，不用 `*` | 🟡 生产环境 |
| 文件上传安全 | 校验 magic bytes（仅图片）限制 5MB、9张，扩展名不可信 | 🟡 建议 |
| 接口限流 | 登录/验证码接口加频率限制（防止暴力破解） | 🟡 建议 |
| 手机号隐私 | 数据库存明文（比赛阶段可接受），API 返回时脱敏 `138****8000` | 🟡 建议 |
| HTTPS | 生产环境用 Let's Encrypt 免费证书 | 🟢 加分项 |
| CSRF | SPA + JWT 模式下天然免疫（非 Cookie 传递 Token） | 🟢 了解即可 |
| 登出安全 | JWT 无状态特性：登出后 token 在过期前仍有效（比赛阶段可接受） | 🟢 标注风险 |

### 11.2 测试策略（一个人怎么做测试）

| 测试类型 | 范围 | 方法 |
|---------|------|------|
| **手动功能测试** | 所有页面和接口 | 对照功能清单逐项点一遍 |
| **AI 回答质量测试** | 智能客服 | 准备 20 个典型问题，看回复是否准确、是否编造 |
| **异常测试** | 关键接口 | 不登录访问需登录接口、传错误参数、断开网络 |
| **兼容性测试** | 浏览器 | Chrome + Edge（同为 Chromium，问题不大） |
| **性能感觉测试** | 页面加载 | 首页/列表页 3 秒内出内容，AI 回复 10 秒内出第一个字 |

> 不需要写自动化测试（单元测试/E2E测试），一个人的项目，手动测足够，写测试的时间够再多做两个功能了。

# backend/utils/external_api.py — 统一的外部API调用封装
import httpx
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
async def call_llm_with_retry(llm, prompt, fallback_llm=None):
    """带重试和降级的LLM调用"""
    try:

    except Exception as e:

            return await fallback_llm.ainvoke(prompt)
        raise

async def fetch_weather_with_fallback(region):
