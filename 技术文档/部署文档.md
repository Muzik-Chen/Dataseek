## 2. 服务器要求

### 最低配置

| 资源 | 最低 | 推荐 |
|------|------|------|
| CPU | 2 核 | 4 核 |
| 内存 | 4 GB | 8 GB |
| 磁盘 | 20 GB | 40 GB+ |
| 系统 | Ubuntu 22.04 LTS | Ubuntu 22.04 LTS |
| 网络 | 可访问外网（调用 LLM API） | 国内服务器需配置 HuggingFace 镜像 |

> ⚠️ 2核4G 运行全部组件（Nginx + FastAPI + MySQL + AI 模型）已经是极限。AI Embedding 模型自身占用约 400MB 内存。如需在更低配服务器运行，可考虑使用外部 MySQL（云数据库）或关闭部分 AI 功能。

### 端口规划

| 端口 | 服务 | 说明 |
|------|------|------|
| 80 | Nginx | HTTP 入口 |
| 443 | Nginx | HTTPS（生产环境必须） |
| 8000 | uvicorn | FastAPI（仅绑定 127.0.0.1，不对外暴露） |
| 3306 | MySQL | 数据库（仅绑定 127.0.0.1） |

## 3. 部署步骤

### 步骤 1：系统基础环境

```bash
# 更新系统
sudo apt update && sudo apt upgrade -y

# 安装编译工具（sentence-transformers / faiss-cpu 需要 C++ 编译器 + BLAS 库）
sudo apt install build-essential libopenblas-dev -y

# 安装 Python 3.12
sudo add-apt-repository ppa:deadsnakes/ppa -y
sudo apt install python3.12 python3.12-venv python3.12-dev -y

# 安装 Node.js 22（用于前端构建）
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install nodejs -y

# 安装 Nginx
sudo apt install nginx -y

# 安装 MySQL 8.0
sudo apt install mysql-server-8.0 -y
sudo systemctl enable mysql
sudo systemctl start mysql

# 安全初始化（设置 root 密码、删除匿名用户等）
sudo mysql_secure_installation
```

### 步骤 2：创建项目目录

```bash
# 创建项目根目录
sudo mkdir -p /var/www/chaoshan-platform
sudo chown $USER:$USER /var/www/chaoshan-platform

# 拷贝项目文件到服务器（选其一）
# 方式 A：Git 克隆
cd /var/www
git clone <你的仓库地址> chaoshan-platform

# 方式 B：scp 上传
# 在本地执行: scp -r ./chaoshan-platform/* user@服务器IP:/var/www/chaoshan-platform/
```

> ⚠️ 无论哪种方式，务必确保 `data/` 目录（含 `init_db.sql` 和所有 `seed_*.sql`）和 `backend/ai/prompts/` 以及 `backend/ai/knowledge/` 目录已完整上传。这些是运行时必需的数据文件。

### 步骤 3：数据库初始化

```bash
# 登录 MySQL
sudo mysql -u root

# 创建数据库（字符集必须 utf8mb4 以支持 emoji）
CREATE DATABASE IF NOT EXISTS chaoshan_platform
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

# 创建专用用户（安全最佳实践：不用 root 跑应用）
CREATE USER IF NOT EXISTS 'chaoshan_user'@'localhost'
  IDENTIFIED BY '替换为你的强密码';

# 授权
GRANT ALL PRIVILEGES ON chaoshan_platform.* TO 'chaoshan_user'@'localhost';
FLUSH PRIVILEGES;
EXIT;

# 导入种子数据（演示用：美食、非遗、酒店、人流等）
mysql -u chaoshan_user -p chaoshan_platform < /var/www/chaoshan-platform/data/seed_data.sql
mysql -u chaoshan_user -p chaoshan_platform < /var/www/chaoshan-platform/data/seed_hotels.sql
mysql -u chaoshan_user -p chaoshan_platform < /var/www/chaoshan-platform/data/seed_coordinates.sql
mysql -u chaoshan_user -p chaoshan_platform < /var/www/chaoshan-platform/data/seed_crowd_locations.sql
```

> **表结构由应用自动创建** — FastAPI 启动时 `init_db()` 会读取 `data/init_db.sql` 并执行 DDL（跳过已存在的表，幂等安全）。不需要手动导入 `init_db.sql`。

### 步骤 6：预下载 AI Embedding 模型

```bash
cd /var/www/chaoshan-platform/backend
source venv/bin/activate

# 预下载 BGE 中文 Embedding 模型（~400MB）
# 国内服务器务必设置 HF_ENDPOINT 镜像，否则下载极慢或失败
python -c "
import os
os.environ['HF_ENDPOINT'] = 'https://hf-mirror.com'
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('BAAI/bge-small-zh-v1.5')
print('模型下载完成')
"
```

> ⚠️ **为什么必须预下载？** 应用代码中 `embeddings.py` 设置了 `local_files_only=True`，这是为了防止首次启动时网络不稳定导致 SentenceTransformer 阻塞数分钟。如果模型未预下载，应用会启动但 **RAG（知识库检索增强生成）将被静默禁用**，智能客服将降级为纯 LLM 模式（没有知识库支持，可能编造内容）。

模型下载后在 HuggingFace 缓存目录（通常 `~/.cache/huggingface/hub/`），后续启动直接加载本地缓存。

### 步骤 9：配置 Nginx

```bash
# 创建站点配置
sudo nano /etc/nginx/sites-available/chaoshan
```

写入以下内容：

```nginx
server {
    listen 80;
    server_name 你的域名或IP;

    # 前端静态文件
    location / {
        root /var/www/chaoshan-platform/frontend/dist;
        index index.html;
        try_files $uri $uri/ /index.html;  # Vue Router History 模式
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE 流式输出必须关闭缓冲
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;  # AI 行程生成 SSE 可能需 2-3 分钟
    }

    # 用户上传图片（30 天缓存）
    location /uploads/ {
        alias /var/www/chaoshan-platform/backend/uploads/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 静态资源（7 天缓存）
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
        root /var/www/chaoshan-platform/frontend/dist;
        expires 7d;
        add_header Cache-Control "public, immutable";
    }

    # 日志
    access_log /var/log/nginx/chaoshan-access.log;
    error_log /var/log/nginx/chaoshan-error.log;
}
```

启用并验证：

```bash
# 创建软链接启用站点
sudo ln -s /etc/nginx/sites-available/chaoshan /etc/nginx/sites-enabled/

# 删除默认站点（避免冲突）
sudo rm -f /etc/nginx/sites-enabled/default

# 测试 Nginx 配置语法
sudo nginx -t

# 重载 Nginx
sudo systemctl reload nginx
```

### 步骤 10：配置 systemd 服务

```bash
# 创建服务文件
sudo nano /etc/systemd/system/chaoshan-backend.service
```

写入以下内容：

```ini
[Unit]
Description=Chaoshan Platform Backend
After=network.target mysql.service
Wants=mysql.service

[Service]
Type=simple
User=www-data
WorkingDirectory=/var/www/chaoshan-platform/backend
EnvironmentFile=/var/www/chaoshan-platform/backend/.env
ExecStart=/var/www/chaoshan-platform/backend/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8000 --workers 1
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

> ⚠️ **为什么用 --workers 1 而不是 2？**
>
> uvicorn 的 `--workers` 参数会 fork 多个独立进程。每个 worker 进程各自执行完整的 lifespan 启动逻辑：
> - **APScheduler 重复启动**：2 个 worker = 2 份定时任务调度器 → 天气数据每小时拉取 2 次
> - **AI 模型重复加载**：2 个 worker = Embedding 模型加载 2 份到内存 → ~400MB × 2 = 800MB+
> - **init_db() 并发执行**：2 个 worker 同时尝试建表 → 可能产生竞争条件
> - **OOM 风险**：在 2核4G 服务器上，2 个 worker 的内存占用几乎肯定会触发 OOM Killer
>
> 比赛演示阶段，单 worker 完全够用。如需提高并发能力，推荐迁移到 `gunicorn + uvicorn.workers.UvicornWorker`，使用 `--preload` 在 fork 前加载 AI 模型实现内存共享。

启动并验证：

```bash
sudo systemctl daemon-reload
sudo systemctl enable chaoshan-backend

# 步骤11（首次启动验证）完成后再 start
# sudo systemctl start chaoshan-backend
```

### 步骤 11：首次启动验证

**先以前台模式运行，观察初始化日志：**

```bash
cd /var/www/chaoshan-platform/backend
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8000
```

**正常启动日志应包含以下所有行（按顺序）：**

```
[DB] SQL 文件不存在: .../data/init_db.sql — 如果路径错误
     或
[DB] 数据库初始化完成（skip: ... 表示已存在的表被跳过）

[AI] Embedding 模型加载完成          ← 必须出现，否则 RAG 被禁用
[AI] FAISS 知识库加载完成            ← 必须出现，否则智能客服无知识库
[AI] RAG Chain 初始化完成
[AI] TripAgent 初始化完成

[Scheduler] APScheduler 已启动

[Cleanup] 已将 0 个过期会话标记为 closed
```

**常见初始化失败场景及处理：**

| 日志 | 原因 | 处理 |
|------|------|------|
| `[AI] Embedding 模型加载失败` | 模型未预下载 | 回到步骤 6，确保模型下载成功 |
| `[AI] FAISS 知识库不可用` | `ai/knowledge/*.json` 文件缺失或格式错误 | 检查文件完整性，确认 JSON 格式正确 |
| `[DB] SQL 文件不存在` | `data/` 目录未部署或路径错误 | 确认 `data/init_db.sql` 相对于 `backend/` 的路径为 `../data/init_db.sql` |
| 数据库连接失败 | `.env` 配置错误或 MySQL 未启动 | `sudo systemctl status mysql`，检查 `.env` 中 `DB_*` 变量 |
| `[Scheduler] 启动失败` | 和风天气 API Key 无效或网络问题 | 不影响核心功能，天气/人流功能降级 |

**全部日志确认无误后，Ctrl+C 停止前台进程，改为 systemd 启动：**

```bash
sudo systemctl start chaoshan-backend
sudo systemctl status chaoshan-backend
```

## 5. AI 模块初始化

### 初始化流程

应用启动时按以下顺序自动初始化 AI 模块：

```
启动
  │
  ├─ 1. init_db()                从 data/init_db.sql 建表（幂等，已存在的表跳过）
  │
  ├─ 2. get_embedding_model()    加载 BAAI/bge-small-zh-v1.5（384维，~400MB）
  │   │                           失败 → RAG 降级为纯 LLM 模式
  │   │
  ├─ 3. load_knowledge_base()    从 ai/knowledge/*.json + *.md 构建 FAISS 索引
  │   │                           首次运行：构建索引（30-60s）→ 持久化到 ai/faiss_index/
  │   │                           后续运行：直接加载已有索引（秒级）
  │   │
  ├─ 4. init_rag_chain()         创建 LangChain RAG 检索链
  │   │
  └─ 5. get_trip_app()           创建 LangGraph 行程规划 StateGraph
```

### FAISS 索引管理

```bash
# 索引文件位置
ls -lh backend/ai/faiss_index/

```

## 7. 常见问题与排查

### 7.1 502 Bad Gateway

```
原因：Nginx 无法连接后端 FastAPI
排查：
  sudo systemctl status chaoshan-backend   # 后端是否在运行？
  sudo journalctl -u chaoshan-backend -f   # 看看启动日志
  ss -tlnp | grep 8000                     # 8000 端口是否在监听？
```

### 7.2 AI 模型加载失败 / 智能客服不回复

```
症状：[AI] Embedding 模型加载失败（RAG 将降级为纯 LLM 模式）
原因：BGE 模型未下载或 local_files_only=True 阻止了在线下载

排查：
  # 检查模型是否已缓存
  ls ~/.cache/huggingface/hub/models--BAAI--bge-small-zh-v1.5/

  # 如果没有 → 手动下载（步骤 6）
  python -c "
  import os
  os.environ['HF_ENDPOINT'] = 'https://hf-mirror.com'
  from sentence_transformers import SentenceTransformer
  SentenceTransformer('BAAI/bge-small-zh-v1.5')
  "

  # 下载后重启
  sudo systemctl restart chaoshan-backend
```

### 7.3 行程规划一直转圈 / 超时

```
原因：LLM API 调用失败，或模型后备链中所有模型都不可用

排查：
  # 检查 API Key 是否正确
  cat /var/www/chaoshan-platform/backend/.env | grep API_KEY

  # 检查 LLM Provider 配置
  cat /var/www/chaoshan-platform/backend/.env | grep LLM_PROVIDER

  # 查看后端日志
  sudo journalctl -u chaoshan-backend -f | grep -i "error\|fail\|timeout"

  # 手动测试 API 连通性（以智谱为例）
  curl https://open.bigmodel.cn/api/paas/v4/chat/completions \
    -H "Authorization: Bearer $ZHIPU_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{"model":"glm-4-flash","messages":[{"role":"user","content":"你好"}]}'
```

### 7.4 图片上传失败

```
原因：uploads 目录权限不足

修复：
  sudo chown www-data:www-data /var/www/chaoshan-platform/backend/uploads
  sudo chmod 755 /var/www/chaoshan-platform/backend/uploads

  # 确认 .env 中 UPLOAD_DIR 为绝对路径
  grep UPLOAD_DIR /var/www/chaoshan-platform/backend/.env
  # 应输出: UPLOAD_DIR=/var/www/chaoshan-platform/backend/uploads
```

### 7.5 验证码邮件收不到

```
排查：
  # 检查 .env 中 SMTP 配置
  cat /var/www/chaoshan-platform/backend/.env | grep SMTP_

  # 确认 SMTP_PASSWORD 是 163 邮箱的"授权码"，不是登录密码
  # 确认 SMTP_FROM 和 SMTP_USERNAME 一致

  # 查看邮件发送日志
  sudo journalctl -u chaoshan-backend -f | grep "\[EMAIL\]"

  # 常见输出：
  # [EMAIL] 发送成功 → 目标邮箱   → 正常
  # [EMAIL] 发送失败 → 目标邮箱: Authentication failed → 授权码错误
```

### 7.6 MySQL 连接失败

```
症状：启动日志显示 Can't connect to MySQL server

排查：
  sudo systemctl status mysql        # MySQL 是否运行？
  mysql -u chaoshan_user -p -h localhost chaoshan_platform  # 能否登录？
  cat /var/www/chaoshan-platform/backend/.env | grep DB_    # 配置正确？
```

### 7.7 页面刷新后 404

```
原因：Nginx try_files 未配置或配置错误

检查 Nginx 配置中是否有：
  try_files $uri $uri/ /index.html;

如果没有 → 补充后 sudo systemctl reload nginx
```

## 9. 安全注意事项

### 必须做到

- [ ] `.env` 中的 `JWT_SECRET` 已替换为 `.env.example` 默认值之外的强随机字符串
- [ ] 所有 API Key（ZHIPU / DEEPSEEK / QWEN）使用真实密钥，不在代码仓库中硬编码
- [ ] `SMTP_PASSWORD` 使用 163 邮箱的授权码（非登录密码）
- [ ] `DEBUG=false`（生产环境）
- [ ] `CORS_ORIGINS` 仅包含实际域名（不要用 `*`）
- [ ] MySQL 用户 `chaoshan_user` 仅授予 `chaoshan_platform` 数据库权限（不要 GRANT ALL ON *.*）
- [ ] uvicorn 绑定 `127.0.0.1` 而非 `0.0.0.0`（不对外暴露后端端口）

### 强烈建议

- [ ] 配置 HTTPS（Let's Encrypt 免费证书 + certbot）
- [ ] 配置 UFW 防火墙（仅开放 80/443）
- [ ] 定期更新系统：`sudo apt update && sudo apt upgrade`
- [ ] 设置 MySQL root 密码且不用于应用连接
- [ ] 在前端构建时移除 `VITE_AMAP_KEY` 等在前端暴露的 Key（高德 Key 前端使用，无法隐藏，但可配置 Web 端域名白名单限制）

### 比赛阶段可接受

- [ ] JWT 登出无服务端失效机制（无状态 JWT 的固有限制，7 天过期）
- [ ] 手机号数据库明文存储（生产环境建议加密或脱敏）
- [ ] 无接口限流（生产环境建议 Nginx `limit_req` 或 FastAPI middleware）
- [ ] 无 CDN / 静态资源 OSS 分离
